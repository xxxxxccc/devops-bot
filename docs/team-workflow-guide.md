## 系统简介

Filo AI Workflow 是一个部署在 EC2 上的 AI 编码助手。团队成员在**飞书群聊**中用自然语言描述需求，AI 会自动理解意图、分析代码、执行修改，并在 **GitHub** 上提交 Draft PR。

**核心流程：**

```
飞书群聊发消息 → AI 分析意图 → 自动执行代码修改 → GitHub Draft PR → 人工 Review → 合并
```

**支持的能力：**

| 能力 | 说明 |
|------|------|
| 日常对话 | 回答技术问题、讨论方案、查阅代码 |
| 代码任务 | 修 Bug、改文案、新功能、重构 |
| PR 审查 | AI 对 PR 进行逐行代码评审 |
| PR 修改 | 根据 Review 反馈或新需求直接推送修改到已有 PR |
| Issue 管理 | 自动创建、扫描、执行 GitHub Issue |
| 项目记忆 | 记住历史决策、上下文、过去的任务结果 |

## 快速上手

### 1. 加入飞书群

向管理员获取已绑定 AI Bot 的飞书群聊邀请。

### 2. 确认 Workspace 已注册

首次使用时，管理员需在群聊中注册 Workspace：

```
add workspace https://github.com/FiloAI/filo-ai-workspace
```

注册后，AI 即可感知所有子项目（filo-desktop、filo-ios、filo-android、FiloMailCenter 等），并根据消息内容自动路由到正确的仓库。

### 3. 开始对话

在群里 **@Bot** 并用自然语言说你的需求即可。

## 与 AI 对话

### 日常对话与咨询

像跟同事聊天一样提问，AI 会检索代码和记忆来回答。

**示例：**

```
filo-desktop 的主题切换逻辑在哪里实现的？
```

```
FiloMailCenter 的 SendAs 缓存机制是怎么工作的？
```

```
filo-ios 目前支持哪些多语言？
```

AI 会使用代码搜索工具（grep、read_file 等）查找答案，而不是猜测。

### 查询项目记忆

AI 会记住所有历史任务和决策，可以直接查询。

**示例：**

```
上次 filo-desktop 的搜索功能修改了哪些内容？
```

```
之前讨论的草稿本地保存方案最终决定了什么？
```

### 创建开发任务

描述你需要的修改，AI 会评估风险后自动选择合适的执行策略。

**示例（低风险，直接执行）：**

```
帮我把 filo-desktop 设置页面的"退出登录"按钮文案改成"退出账号"
```

**示例（中等风险，需要审批）：**

```
给 filo-android 添加一个应用内更新的弹窗功能，展示更新说明和确认按钮
```

**示例（高风险，仅创建讨论 Issue）：**

```
我们要不要把 FiloMailCenter 的数据库从 MySQL 迁移到 PostgreSQL？
```

### PR 代码审查

让 AI 审查指定的 PR。

**示例：**

```
review PR #35
```

```
帮我看看 filo-ios 的 PR #42
```

AI 会分析 diff，给出整体评价和逐行评论，结果同时发送到群聊和 GitHub PR 评论区。

### 修改已有 PR

对已有的 PR 追加修改，AI 会 checkout 该 PR 分支，应用修改后 push。

**示例：**

```
修改 PR #23，把错误提示从英文改成中文
```

```
PR #45 的 code review 有几个 warning，帮我处理一下
```

## 三级任务体系

AI 会根据风险评估自动选择合适的执行层级：

### Tier 1：直接执行

**适用场景：** 范围明确、影响小、可轻松回退

- 文案 / 配置修改
- 简单 Bug 修复
- 样式调整、typo 修复
- 单文件或少量文件变更

**流程：**

```
你说需求 → AI 立即执行 → 创建 Issue 跟踪 → 提交 Draft PR → 群聊通知结果
```

### Tier 2：提案审批

**适用场景：** 涉及新功能、重构、多模块变更

- 新增功能模块
- 跨文件重构
- 添加新依赖
- 接口变更

**流程：**

```
你说需求 → AI 创建 GitHub Issue（含方案描述）
         → 团队在 Issue 下讨论
         → 有人点 👍 / ❤️ / 🎉 表示审批通过
         → AI 自动执行任务 → 提交 Draft PR
```

**如何审批：** 在 GitHub Issue 页面添加 👍、❤️ 或 🎉 reaction 即可。

### Tier 3：创建讨论

**适用场景：** 高风险、模糊需求、需要团队讨论

- 架构变更
- 数据库迁移
- 破坏性 API 变更
- 描述不够明确的需求

**流程：**

```
你说需求 → AI 创建 GitHub Issue（仅供讨论）→ 团队讨论 → 人工决定后续步骤
```

> **提示：** 讨论完成后，如果想让 AI 执行，给 Issue 加上 `filo-ai-workflow` label + 审批 reaction，AI 会在下次轮询时自动拾取并执行。

## Issue 工作流

### AI 自动扫描 Issue

系统每 3 分钟（可配置）自动扫描以下来源的 Issue：

| 来源 | 说明 |
|------|------|
| Bot 创建的 Issue | Tier 2 提案，等待审批 reaction |
| 项目仓库 Issue | 带有 `filo-ai-workflow` label 的外部 Issue |
| Workspace 仓库 Issue | `filo-ai-workspace` 仓库的 Issue，自动分发给对应子项目 |

### 审批与执行

1. 团队成员在 GitHub Issue 页面给 Issue 添加 **👍 / ❤️ / 🎉** reaction
2. AI 在下次轮询时发现审批，读取 Issue 全部讨论内容
3. **Issue AI** 综合讨论，生成清晰的任务描述（过滤掉元讨论，聚焦最终共识）
4. 任务执行，结果以 Draft PR 提交

### 跨仓库任务分发

当 Issue 涉及多个仓库时（通过 Workspace 上下文判断）：

1. AI 先判断该 Issue 应由哪些仓库处理
2. 在每个目标仓库自动创建**子 Issue**（自动审批，无需重复操作）
3. 每个子 Issue 独立执行，生成各自的 PR

**示例：** 在 `filo-ai-workspace` 仓库提 Issue 说"更新所有平台的版本号"，AI 会自动在 filo-desktop、filo-ios、filo-android 各创建一个子 Issue 并分别执行。

### Issue 质量门禁

AI 在执行前会进行质量检查：

- **actionable**：内容清晰可执行 → 进入执行流程
- **needs_info**：信息不足 → 评论要求补充，有新更新后重新评估
- **reject**：内容无意义或不适合自动处理 → 评论说明原因，标记跳过

## PR 审查工作流

### 自动审查 + 自动修复

当 AI 执行任务并创建 PR 后（需开启 `ENABLE_SELF_REVIEW`）：

```
任务完成 → 自动 Review PR
         → 发现 Critical/Warning 问题
         → 自动修复 + push 到同一分支
         → 重新 Review
         → 最多 2 轮自动修复
```

修复后的 PR 包含完整的修改记录，方便人工最终审核。

### 手动触发审查

在飞书群聊中发送：

```
review PR #35
```

审查结果会同时发送到：
- 飞书群聊（摘要）
- GitHub PR 评论区（详细逐行评论）

### 外部 PR 自动扫描

系统默认每 3 分钟扫描所有注册项目的 Open PR 并进行代码审查（通过 `REVIEW_TRIGGER_MODE=polling` 配置）。审查结果仅发送到 GitHub PR 评论区。

已审查过的 PR 不会重复审查（基于 HEAD commit SHA 去重），除非有新的 push。

## Workspace 模式

当前部署使用 Workspace 模式，`filo-ai-workspace` 仓库作为所有项目的入口。

**Workspace 包含：**

- `workspace.json` — 列出所有子项目（ID、Git URL、分支、语言、描述）
- `CLAUDE.md` — 组织级开发规范和项目关系说明

**已注册的项目（示例）：**

| 项目 | 语言 | 描述 |
|------|------|------|
| filo-desktop | TypeScript | AI 邮件桌面客户端 (Electron) |
| filo-ios | Swift | Filo iOS 邮件客户端 |
| filo-android | Kotlin | AI 邮件 Android 客户端 |
| FiloMailCenter | Kotlin | 邮件服务中心 (后端) |

AI 会根据消息内容自动判断目标仓库，无需手动指定。如果判断不准确，可以在消息中明确提及项目名。

**子项目按需克隆：** 首次对某个子项目下达任务时，系统会自动 clone 该仓库。已有的仓库会自动 sync 最新代码。

## 安全机制

| 机制 | 说明 |
|------|------|
| **沙箱隔离** | 每个任务在独立的 Git Worktree 沙箱中执行，互不干扰 |
| **Draft PR** | 所有代码变更以 Draft PR 提交，**必须人工 Review 后才能合并** |
| **危险命令拦截** | `rm -rf /`、`sudo`、`shutdown` 等危险命令被自动屏蔽 |
| **分支保护** | 不会 force push 或删除受保护分支 |
| **三级风险评估** | AI 根据影响范围自动决定是直接执行、等待审批还是仅创建讨论 |
| **记忆命名空间隔离** | 任务记忆和审查记忆分开存储，避免交叉污染 |
| **GitHub App 认证** | 使用 GitHub App 进行安全认证，无需共享个人 Token |

**核心原则：AI 生成的所有代码变更都停留在工作分支上，由人工审核后才会合并到主分支。**

## 常见问题 FAQ

### Q: 我需要 @Bot 才能触发吗？

需要。在飞书群聊中 **@Bot** 发送消息，Bot 才会响应。普通群聊消息不会触发 AI。

### Q: AI 会直接合并代码到主分支吗？

不会。所有变更以 **Draft PR** 形式提交，必须通过人工 Review 后手动合并。

### Q: 提了一个任务但 AI 觉得风险太高，怎么办？

AI 会创建一个 GitHub Issue 供讨论。讨论完成后：
1. 给 Issue 添加 `filo-ai-workflow` label
2. 添加 👍 reaction 表示审批
3. AI 会在下次轮询时自动执行

### Q: PR 审查结果在哪里看？

- 从群聊触发的：**飞书群 + GitHub PR 评论区**
- 自动扫描的：仅 **GitHub PR 评论区**

### Q: 同一个 PR 会被重复审查吗？

不会。系统基于 commit SHA 去重，只有新 push 后才会重新审查。

### Q: 想让 AI 修改已有的 PR 怎么说？

直接在群里说：

```
修改 PR #23，xxxxx（你的修改需求）
```

AI 会 checkout 该 PR 分支，执行修改后 push 到同一分支。

### Q: 跨仓库的需求怎么提？

直接描述需求即可，AI 会根据 Workspace 上下文自动判断涉及哪些仓库。如果需要同时修改多个仓库，AI 会在每个目标仓库分别创建 Issue 和 PR。

也可以直接在 `filo-ai-workspace` 仓库创建 Issue 并加上 `filo-ai-workflow` label，AI 会自动分发。

### Q: AI 执行错了怎么办？

- PR 是 Draft 状态，直接 Close 即可，不会影响主分支
- 可以在群里说"修改 PR #xx，需要改成 xxx"让 AI 修正
- 也可以手动修改后 push

### Q: 审批轮询间隔是多久？

默认 3 分钟。Issue 审批和 PR 审查使用同一轮询周期。如需调整可联系管理员修改配置。

### Q: 支持哪些语言？

AI 会根据你的消息语言自动回复对应语言。中文消息回复中文，英文消息回复英文。PR 审查和 Issue 评论也遵循相同规则。
